## 1. 生产者-消费者模式 (event_dispatcher)

### 在哪里使用
**文件**: `include/mcp_server.h` (第43-165行)

### 具体实现代码

```cpp
class event_dispatcher {
private:
    mutable std::mutex m_;
    std::condition_variable cv_;
    std::atomic<int> id_{0};
    std::atomic<int> cid_{-1};
    std::string message_;
    std::atomic<bool> closed_{false};
    std::chrono::steady_clock::time_point last_activity_{std::chrono::steady_clock::now()};

public:
    // 消费者: 等待事件
    bool wait_event(httplib::DataSink* sink, 
                    const std::chrono::milliseconds& timeout = ...) {
        std::unique_lock<std::mutex> lk(m_);
        bool result = cv_.wait_for(lk, timeout, [...] {
            return cid_.load(std::memory_order_relaxed) == id || 
                   closed_.load(std::memory_order_acquire);
        });
        if (!message_.empty()) {
            message_copy.swap(message_);
            sink->write(message_copy.data(), message_copy.size());
        }
        return true;
    }

    // 生产者: 发送事件
    bool send_event(const std::string& message) {
        std::lock_guard<std::mutex> lk(m_);
        message_ = message;
        cid_.store(id_.fetch_add(1, std::memory_order_relaxed), 
                   std::memory_order_relaxed);
        cv_.notify_one(); // 通知消费者
        return true;
    }
};
```

### 为什么使用这个模式
- **SSE (Server-Sent Events)**: 服务器需要向客户端推送事件消息
- **线程安全**: 使用 `std::mutex` 和 `std::condition_variable` 确保线程安全
- **高效等待**: 消费者线程在无消息时阻塞，避免CPU空转

### 优点
1. **解耦生产者和消费者**: 生产者(服务器业务逻辑)和消费者(HTTP响应)不直接依赖
2. **支持超时机制**: 防止消费者无限期等待
3. **支持关闭信号**: 通过 `closed_` 原子变量支持优雅关闭
4. **活动追踪**: 记录最后活动时间用于会话超时管理

## 2. 观察者模式 (Observer)

### 在哪里使用
**文件**: `include/mcp_resource.h` (第234-305行) - `resource_manager` 类

### 具体实现代码

```cpp
class resource_manager {
private:
    // 订阅者存储: 订阅ID -> (资源URI, 回调函数)
    std::map<int, std::pair<std::string, 
        std::function<void(const std::string&)>>> subscriptions_;
    int next_subscription_id_ = 1;

public:
    // 订阅资源变化
    int subscribe(const std::string& uri, 
                  std::function<void(const std::string&)> callback) {
        int id = next_subscription_id_++;
        subscriptions_[id] = std::make_pair(uri, callback);
        return id;
    }

    // 取消订阅
    bool unsubscribe(int subscription_id) {
        auto it = subscriptions_.find(subscription_id);
        if (it == subscriptions_.end()) return false;
        subscriptions_.erase(it);
        return true;
    }

    // 通知观察者(资源变化时)
    void notify_resource_changed(const std::string& uri) {
        for (const auto& [id, sub] : subscriptions_) {
            if (sub.first == uri) {
                sub.second(uri); // 调用观察者回调
            }
        }
    }
};
```

### 为什么使用这个模式
- **资源变化通知**: 当资源被修改时，需要通知所有订阅了该资源的客户端
- **解耦**: 资源管理器和观察者之间不需要直接依赖，通过回调函数解耦

### 优点
1. **松耦合**: 观察者和主题(资源)之间不需要直接引用
2. **动态订阅**: 客户端可以随时订阅/取消订阅资源变化
3. **一对多通知**: 一个资源变化可以通知多个观察者

```cpp
// 类型定义: 将函数作为策略
using method_handler = std::function<json(const json&, const std::string&)>;
using tool_handler = method_handler;
using notification_handler = std::function<void(const json&, const std::string&)>;
using auth_handler = std::function<bool(const std::string&, const std::string&)>;

// 注册策略(处理器)
void register_method(const std::string& method, method_handler handler);
void register_notification(const std::string& method, notification_handler handler);
void register_tool(const tool& tool, tool_handler handler);

// 内部存储: 方法名 -> 策略(处理函数)
std::map<std::string, method_handler> method_handlers_;
std::map<std::string, notification_handler> notification_handlers_;
std::map<std::string, std::pair<tool, tool_handler>> tools_;

// 使用策略
json server::process_request(const request& req, const std::string& session_id) {
    auto it = method_handlers_.find(req.method);
    if (it != method_handlers_.end()) {
```

### 为什么使用这个模式
- **可扩展性**: 用户可以注册自定义的方法处理器
- **运行时绑定**: 处理器在运行时确定，而不是编译时
- **开闭原则**: 新增方法不需要修改服务器核心代码

### 优点
1. **高度灵活**: 可以在运行时注册/替换处理策略
2. **解耦**: 业务逻辑与核心协议处理分离
3. **易于测试**: 可以为每个handler编写独立测试

## 4. 模板方法模式 (Template Method)

### 在哪里使用
**文件**: `src/mcp_server.cpp` - `process_request()` 方法

### 具体实现代码

```cpp
json server::process_request(const request& req, const std::string& session_id) {
    // 模板方法骨架: 定义了处理请求的整体流程
    
    // 步骤1: 检查是否为通知
    if (req.is_notification()) {
        if (req.method == "notifications/initialized") {
            set_session_initialized(session_id, true);
        }
        return json::object();
    }
    
    // 步骤2: 处理特殊方法
    if (req.method == "initialize") {
        return handle_initialize(req, session_id);
    } else if (req.method == "ping") {
        return response::create_success(req.id, json::object()).to_json();
    }

    // 步骤3: 验证会话状态
    if (!is_session_initialized(session_id)) {
        return response::create_error(req.id, error_code::invalid_request, 
                                      "Session not initialized").to_json();
    }
    
    // 步骤4: 查找并调用handler
    method_handler handler = find_handler(req.method);
    if (handler) {
        json result = handler(req.params, session_id);
        return response::create_success(req.id, result).to_json();
    }
    
    // 步骤5: 处理错误
    return response::create_error(req.id, error_code::method_not_found, 
                                  "Method not found").to_json();
}
```

### 为什么使用这个模式
- **统一处理流程**: 所有请求都遵循相同的处理骨架
- **代码复用**: 共同的错误处理、会话验证逻辑被复用
- **易于扩展**: 可以在骨架的各步骤中插入新的处理逻辑

### 优点
1. **代码复用**: 公共逻辑集中在一处
2. **易于维护**: 流程清晰，易于理解和修改
3. **可覆盖性**: 特定步骤可以被子类或handler覆盖

```cpp
class tool_builder {
private:
    std::string name_;
    std::string description_;
    json parameters_;
    json annotations_;
    std::vector<std::string> required_params_;

public:
    explicit tool_builder(const std::string& name) : name_(name) {}
    
    // 链式调用 (Fluent API)
    tool_builder& with_description(const std::string& description) {
        description_ = description;
        return *this;
    }
    
    tool_builder& with_string_param(const std::string& name, 
                                     const std::string& description, 
                                     bool required = true) {
        return add_param(name, description, "string", required);
    }
    
    tool_builder& with_number_param(const std::string& name, 
                                     const std::string& description, 
                                     bool required = true) {
        return add_param(name, description, "number", required);
    }
    
    // 工厂方法: 最终构建产品
    tool build() const {
        tool t;
        t.name = name_;
        t.description = description_;
        
        json schema = parameters_;
        schema["type"] = "object";
        if (!required_params_.empty()) {
            schema["required"] = required_params_;
        }
        
        t.parameters_schema = schema;
        t.annotations = annotations_;
        return t;
    }
};

// 使用示例
tool my_tool = tool_builder("my_tool")
    .with_description("A sample tool")
    .with_string_param("input", "Input parameter", true)
    .with_number_param("count", "Count parameter", false)
    .build();
```

### 为什么使用这个模式
- **简化复杂对象创建**: 工具对象有多个属性，建造者模式使创建过程清晰
- **链式API**: 提供流畅的接口，易于阅读和使用
- **参数验证**: 可以在build()中进行最终验证

### 优点
1. **代码清晰**: 创建复杂对象的代码更易读
2. **不可变对象**: build()返回后，对象完全初始化
3. **易于扩展**: 可以添加新的可选参数而不破坏现有代码

```cpp
public:
    explicit auto_lock(std::mutex& mutex) : lock_(mutex) {}  // 构造函数中获取锁
    // 析构函数中自动释放锁
```

### 为什么使用这个模式
- **异常安全**: 即使发生异常，锁也会在作用域结束时自动释放

2. **减少错误**: 避免忘记释放锁导致的死锁
3. **代码简洁**: 减少显式的锁管理代码

## 8. 线程池模式 (Thread Pool)

### 在哪里使用
**文件**: `include/mcp_thread_pool.h`

### 具体实现代码

```cpp
class thread_pool {
private:
    size_t min_threads_;
    size_t max_threads_;
    std::chrono::milliseconds idle_timeout_;
    
    std::atomic<size_t> active_threads_{0};
    size_t idle_threads_{0};
    
    std::queue<std::function<void()>> tasks_;
    std::mutex queue_mutex_;
    std::condition_variable condition_;

public:
    // 动态线程池: 支持自动扩缩容
    template<class F, class... Args>
    auto enqueue(F&& f, Args&&... args) -> std::future<...> {
        auto task = std::make_shared<std::packaged_task<return_type()>>(
            std::bind(std::forward<F>(f), std::forward<Args>(args)...)
        );
        
        std::future<return_type> result = task->get_future();
        
        {
            std::unique_lock<std::mutex> lock(queue_mutex_);
            tasks_.emplace([task]() { (*task)(); });
            
            // 自动扩容: 无空闲线程且未达上限则创建新线程
            if (idle_threads_ == 0 && active_threads_ < max_threads_) {
                spawn_thread();
            }
        }
        
        condition_.notify_one();
        return result;
    }

private:
    void worker_loop() {
        while (true) {
            bool is_timeout = !condition_.wait_for(lock, idle_timeout_, [this] { 
                return stop_ || !tasks_.empty(); 
            });
            
            // 自动缩容: 超时且有多余线程则退出
            if (is_timeout && tasks_.empty() && active_threads_ > min_threads_) {
                active_threads_--;
                break;
            }
            
            task = std::move(tasks_.front());
            tasks_.pop();
        }
        task();
    }
};
```

### 为什么使用这个模式
- **高并发处理**: 服务器需要处理大量并发请求
- **资源优化**: 根据负载自动调整线程数量
- **异步处理**: 通知类请求可以异步处理，不阻塞主线程

### 优点
1. **性能优化**: 线程数量根据负载动态调整
2. **资源控制**: 避免创建过多线程导致资源耗尽
3. **任务队列**: 解耦任务提交和任务执行

## 9. 代理/装饰器模式 (Decorator)

### 在哪里使用
**文件**: `include/mcp_server.h` - `make_async_handler` 模板方法 (第479-486行)

### 具体实现代码

```cpp
// 将同步handler包装为异步handler
template<typename F>
std::function<std::future<json>(const json&, const std::string&)> 
make_async_handler(F&& handler) {
    return [handler = std::forward<F>(handler)](
        const json& params, const std::string& session_id) 
        -> std::future<json> {
        return std::async(std::launch::async, [handler, params, session_id]() -> json {
            return handler(params, session_id);
        });
    };
}
```

### 为什么使用这个模式
- **异步执行**: 将同步函数转换为异步执行
- **不修改原函数**: 包装器不改变原handler的接口

### 优点
1. **透明转换**: 调用方无需知道handler是同步还是异步
2. **功能增强**: 在不修改原对象的情况下添加新功能
