## 1. 什么是单元测试

### 定义
**单元测试 (Unit Testing)** 是对软件中的最小可测试单元进行检验的测试方法。

### 特点
| 特性 | 说明 |
|------|------|
| **独立性** | 每个测试用例独立运行，不依赖其他测试 |
| **可重复** | 相同的测试可以重复运行，结果一致 |
| **自动化** | 可自动执行，无需人工干预 |
| **快速** | 运行时间短，通常毫秒级 |

### 示例对比

```cpp
// ❌ 没有单测：手动测试
void test_manual() {
    request req = request::create("test_method", {{"key", "value"}});
    json req_json = req.to_json();
    
    // 手动打印检查
    std::cout << req_json << std::endl;
    
    // 如果有 100 个功能，每个都要手动检查...
}

// ✅ 有单测：自动验证
TEST_F(MessageFormatTest, RequestMessageFormat) {
    request req = request::create("test_method", {{"key", "value"}});
    json req_json = req.to_json();
    
    // 自动断言
    EXPECT_EQ(req_json["jsonrpc"], "2.0");
    EXPECT_EQ(req_json["method"], "test_method");
}
```

## 4. 现有单测解析

### 4.1 测试分类

本项目单测分为 **4 个测试环境**，每个环境针对不同功能：

| 环境类 | 测试内容 | 端口 |
|--------|----------|------|
| `LifecycleEnvironment` | 生命周期/初始化流程 | 8080 |
| `VersioningEnvironment` | 版本控制 | 8081 |
| `PingEnvironment` | Ping 心跳机制 | 8082 |
| `ToolsEnvironment` | 工具注册和调用 | 8083 |

### 4.2 消息格式测试

```cpp
// test/mcp_test.cpp:32-94

// 测试请求消息格式
TEST_F(MessageFormatTest, RequestMessageFormat) {
    request req = request::create("test_method", {{"key", "value"}});
    json req_json = req.to_json();
    
    EXPECT_EQ(req_json["jsonrpc"], "2.0");
    EXPECT_TRUE(req_json.contains("id"));
    EXPECT_EQ(req_json["method"], "test_method");
    EXPECT_EQ(req_json["params"]["key"], "value");
}

// 测试成功响应格式
TEST_F(MessageFormatTest, ResponseMessageFormat) {
    response res = response::create_success("test_id", {{"key", "value"}});
    json res_json = res.to_json();
    
    EXPECT_EQ(res_json["jsonrpc"], "2.0");
    EXPECT_EQ(res_json["id"], "test_id");
    EXPECT_EQ(res_json["result"]["key"], "value");
    EXPECT_FALSE(res_json.contains("error"));
}

// 测试错误响应格式
TEST_F(MessageFormatTest, ErrorResponseMessageFormat) {
    response res = response::create_error(
        "test_id", 
        error_code::invalid_params, 
        "Invalid parameters", 
        {{"details", "Missing required field"}}
    );
    json res_json = res.to_json();
    
    EXPECT_EQ(res_json["error"]["code"], static_cast<int>(error_code::invalid_params));
    EXPECT_EQ(res_json["error"]["message"], "Invalid parameters");
}

// 测试通知消息格式
TEST_F(MessageFormatTest, NotificationMessageFormat) {
    request notification = request::create_notification("test_notification", {{"key", "value"}});
    json notification_json = notification.to_json();
    
    EXPECT_FALSE(notification_json.contains("id"));
    EXPECT_TRUE(notification.is_notification());
}
```

### 4.3 环境设置 (Test Fixture)

```cpp
// 测试环境类 - 用于设置/清理测试所需资源
class LifecycleEnvironment : public ::testing::Environment {
public:
    void SetUp() override {
        // 创建服务器
        server::configuration conf = {.host = "localhost", .port = 8080};
        server_ = std::make_unique<server>(conf);
        server_->set_server_info("TestServer", "1.0.0");
        
        // 设置服务器能力
        json server_capabilities = {
            {"logging", json::object()},
            {"prompts", {{"listChanged", true}}},
            {"resources", {{"subscribe", true}, {"listChanged", true}}},
            {"tools", {{"listChanged", true}}}
        };
        server_->set_capabilities(server_capabilities);
        
        // 启动服务器
        server_->start(false);
        
        // 创建客户端
        client_ = std::make_unique<sse_client>("http://localhost:8080");
    }

    void TearDown() override {
        // 清理资源
        client_.reset();
        server_->stop();
        server_.reset();
    }
    
    static std::unique_ptr<server>& GetServer() { return server_; }
    static std::unique_ptr<sse_client>& GetClient() { return client_; }

private:
    static std::unique_ptr<server> server_;
    static std::unique_ptr<sse_client> client_;
};

// 具体测试类 - 使用环境资源
class LifecycleTest : public ::testing::Test {
protected:
    void SetUp() override {
        client_ = LifecycleEnvironment::GetClient().get();
    }
    sse_client* client_;
};

// 使用测试类
TEST_F(LifecycleTest, InitializeProcess) {
    bool init_result = client_->initialize("TestClient", "1.0.0");
    EXPECT_TRUE(init_result);
    
    json server_capabilities = client_->get_server_capabilities();
    EXPECT_TRUE(server_capabilities.contains("logging"));
    EXPECT_TRUE(server_capabilities.contains("tools"));
}
```

```cpp
// 测试工具注册和调用
class ToolsEnvironment : public ::testing::Environment {
public:
    void SetUp() override {
        // 注册工具
        tool test_tool;
        test_tool.name = "get_weather";
        test_tool.description = "Get current weather information";
        test_tool.parameters_schema = {
            {"type", "object"},
            {"properties", {
                {"location", {
                    {"type", "string"},
                    {"description", "City name"}
                }}
            }},
            {"required", json::array({"location"})}
        };
        
        server_->register_tool(test_tool, [](const json& params, const std::string&) -> json {
            std::string location = params["location"];
            return {
                {"content", json::array({
                    {{"type", "text"}, {"text", "Weather in " + location}}
                })},
                {"isError", false}
            };
        });
        
        // 启动服务...
    }
};

// 测试工具列表
TEST_F(ToolsTest, ListTools) {
    json tools_list = client_->send_request("tools/list").result;
    EXPECT_TRUE(tools_list.contains("tools"));
    EXPECT_EQ(tools_list["tools"][0]["name"], "get_weather");
}

// 测试工具调用
TEST_F(ToolsTest, CallTool) {
    json tool_result = client_->call_tool("get_weather", {{"location", "New York"}});
    EXPECT_TRUE(tool_result.contains("content"));
    EXPECT_EQ(tool_result["content"][0]["text"], "Weather in New York");
}
```

## 5. 如何编写单测

### 5.1 常用断言宏

| 宏 | 说明 | 示例 |
|----|------|------|
| `EXPECT_EQ(a, b)` | 期望 a == b | `EXPECT_EQ(actual, expected)` |
| `EXPECT_NE(a, b)` | 期望 a != b | `EXPECT_NE(x, 0)` |
| `EXPECT_TRUE(a)` | 期望 a 为 true | `EXPECT_TRUE(result)` |
| `EXPECT_FALSE(a)` | 期望 a 为 false | `EXPECT_FALSE(has_error)` |
| `EXPECT_THROW(code, type)` | 期望抛出异常 | `EXPECT_THROW(doit(), std::runtime_error)` |
| `EXPECT_FLOAT_EQ(a, b)` | 浮点数近似相等 | `EXPECT_FLOAT_EQ(x, 3.14)` |
| `ASSERT_*` | 断言失败终止测试 | `ASSERT_EQ(a, b)` |

### 5.2 基本测试用例

```cpp
// 最简单的测试
TEST(TestSuiteName, TestName) {
    // 测试代码
    int result = 1 + 1;
    EXPECT_EQ(result, 2);
}
```

### 5.4 为新功能编写单测

假设要测试一个新的 `mcp_resource` 类：

```cpp
// test/test_resource.cpp

#include <gtest/gtest.h>
#include "mcp_resource.h"

using namespace mcp;

// Test Fixture 模式
class ResourceTest : public ::testing::Test {
protected:
    void SetUp() override {
        // 设置测试环境
    }
    
    void TearDown() override {
        // 清理环境
    }
};

// 测试资源注册
TEST_F(ResourceTest, RegisterResource) {
    auto resource = std::make_shared<resource>("test://resource");
    
    auto& rm = resource_manager::instance();
    rm.register_resource(resource);
    
    auto retrieved = rm.get_resource("test://resource");
    EXPECT_NE(retrieved, nullptr);
    EXPECT_EQ(retrieved->get_uri(), "test://resource");
}

// 测试资源列表
TEST_F(ResourceTest, ListResources) {
    auto resource = std::make_shared<resource>("test://resource");
    resource_manager::instance().register_resource(resource);
    
    json list = resource_manager::instance().list_resources();
    EXPECT_TRUE(list.contains("resources"));
    EXPECT_EQ(list["resources"].size(), 1);
}

// 测试资源不存在
TEST_F(ResourceTest, GetNonExistentResource) {
    auto resource = resource_manager::instance().get_resource("nonexistent://uri");
    EXPECT_EQ(resource, nullptr);
}

// 测试订阅机制
TEST_F(ResourceTest, SubscribeResource) {
    bool notified = false;
    int sub_id = resource_manager::instance().subscribe(
        "test://resource",
        [&notified](const std::string& uri) {
            notified = true;
        }
    );
    
    EXPECT_GT(sub_id, 0);
    
    resource_manager::instance().notify_resource_changed("test://resource");
    EXPECT_TRUE(notified);
}
```

## 6. 如何运行单测

### 6.1 编译单测

```bash
# 1. 进入项目目录
cd E:\c++_mcp_sdk\c++_mcp

# 2. 创建 build 目录并配置 CMake，开启单测选项
mkdir build
cd build
cmake .. -DMCP_BUILD_TESTS=ON

# 3. 编译
cmake --build . --config Release
```

### 6.2 运行所有测试

```bash
# 方式1: 使用 ctest
ctest --output-on-failure

# 方式2: 直接运行测试程序
./test/mcp_tests.exe

# 方式3: 使用 make target (如果使用 Makefile)
make run_tests
```

### 6.3 运行特定测试

```bash
# 运行特定测试套件
./test/mcp_tests.exe --gtest_filter=MessageFormatTest.*

# 运行特定测试用例
./test/mcp_tests.exe --gtest_filter=MessageFormatTest.RequestMessageFormat

# 排除某些测试
./test/mcp_tests.exe --gtest_filter=-LifecycleTest.*
```

### 6.4 常用命令行选项

| 选项 | 说明 |
|------|------|
| `--gtest_filter=pattern` | 按模式过滤测试 |
| `--gtest_list_tests` | 列出所有测试 |
| `--gtest_repeat=3` | 重复运行 3 次 |
| `--gtest_shuffle` | 随机顺序运行 |
| `--gtest_output=json` | 输出 JSON 格式结果 |
| `--gtest_break_on_failure` | 失败时进入调试 |

## 7. 单测的作用与意义

### 7.1 为什么需要单测

```
┌─────────────────────────────────────────────────────────────┐
│                    没有单测的痛苦                            │
├─────────────────────────────────────────────────────────────┤
│  ❌ 修改代码后，不知道是否破坏了现有功能                       │
│  ❌ 手动测试繁琐，容易遗漏                                   │
│  ❌ 难以发现边界条件和异常情况                               │
│  ❌ 代码重构风险极高                                         │
│  ❌ 调试时间远超开发时间                                     │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    有单测的好处                              │
├─────────────────────────────────────────────────────────────┤
│  ✅ 修改代码后，立即知道是否破坏了功能                        │
│  ✅ 自动化，一键运行所有测试                                 │
│  ✅ 各种边界条件被覆盖                                       │
│  ✅ 可以放心重构                                             │
│  ✅ 快速定位问题                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 单测的价值

| 价值 | 说明 |
|------|------|
| **回归测试** | 每次修改后确保现有功能不被破坏 |
| **文档作用** | 测试代码本身就是最好的文档 |
| **设计促进** | 写单测能发现设计问题 |
| **快速反馈** | 开发时立即获得反馈 |
| **代码质量** | 提高代码健壮性和可维护性 |

### 7.3 本项目单测覆盖范围

| 模块 | 测试内容 |
|------|----------|
| `mcp_message` | 请求/响应/通知消息格式 |
| `生命周期` | initialize 初始化流程 |
| `版本控制` | 支持/不支持版本的处理 |
| `Ping` | 心跳机制 |
| `Tools` | 工具注册、列表、调用 |

## 8. 最佳实践

### 8.1 测试原则 (FIRST)

| 原则 | 说明 |
|------|------|
| **F**ast | 测试要快，避免 IO 操作 |
| **I**ndependent | 测试之间不要相互依赖 |
| **R**epeatable | 每次运行结果一致 |
| **S**elf-validating | 测试结果自动判断 |
| **T**imely | 测试应及时编写 |

### 8.2 测试命名规范

```cpp
// 推荐: 描述性的测试名称
TEST_F(ToolsTest, ListTools_ReturnsWeatherTool) { ... }
TEST_F(ToolsTest, CallTool_WithValidParams_Succeeds) { ... }
TEST_F(ToolsTest, CallTool_WithInvalidParams_ReturnsError) { ... }

// 不推荐: 无意义的名称
TEST_F(ToolsTest, Test1) { ... }
TEST_F(ToolsTest, TestTools) { ... }
```

### 8.3 测试结构

```cpp
// 标准的测试结构: AAA (Arrange-Act-Assert)
TEST_F(MyTest, TestCase) {
    // Arrange: 准备测试数据
    auto req = request::create("method", {{"key", "value"}});
    
    // Act: 执行被测试的操作
    json result = server_->process_request(req, session_id);
    
    // Assert: 验证结果
    EXPECT_EQ(result["result"], expected_value);
}
```

### 8.4 常见错误

```cpp
// ❌ 错误1: 测试太多断言
TEST_F(BadTest, TooManyAssertions) {
    // 一个测试应该只测一个概念
    EXPECT_EQ(a, b);
    EXPECT_EQ(c, d);
    EXPECT_EQ(e, f);
    EXPECT_EQ(g, h);  // 太多断言，难以定位问题
}

// ✅ 正确: 每个测试只测一个方面
TEST_F(GoodTest, TestA) { EXPECT_EQ(a, b); }
TEST_F(GoodTest, TestB) { EXPECT_EQ(c, d); }

// ❌ 错误2: 测试依赖执行顺序
TEST_F(BadTest, Test1) { ... }
TEST_F(BadTest, Test2) { 
    // 假设 Test1 已经运行过，这是错误的
}

// ✅ 正确: 每个测试独立
TEST_F(GoodTest, IndependentTest1) { ... }
TEST_F(GoodTest, IndependentTest2) { ... }

// ❌ 错误3: 测试内部创建环境
TEST_F(BadTest, CreatesEnvironmentInside) {
    server s;
    s.start();  // 每个测试都启动服务器，太慢
}

// ✅ 正确: 使用 Test Fixture 共享环境
class Environment : public ::testing::Environment {
    static server s;
};
```

### 8.5 测试覆盖指南

```cpp
// 应该测试的场景：

// 1. 正常路径
TEST_F(MyTest, NormalCase) {
    EXPECT_EQ(add(1, 2), 3);
}

// 2. 边界条件
TEST_F(MyTest, EdgeCase_Zero) {
    EXPECT_EQ(add(0, 5), 5);
}

TEST_F(MyTest, EdgeCase_Negative) {
    EXPECT_EQ(add(-1, 1), 0);
}

// 3. 异常情况
TEST_F(MyTest, InvalidInput) {
    EXPECT_THROW(divide(1, 0), std::runtime_error);
}

// 4. 空/无效状态
TEST_F(MyTest, EmptyInput) {
    EXPECT_TRUE(list_all().empty());
}
```
